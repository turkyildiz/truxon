**REGRESSION FOUND + FIXED 2026-07-23 (DR test):** the nightly offsite rsync had been **silently broken since containerization** — `backup.sh` runs inside the **scheduler container** (docker:cli/Alpine) which ships ssh but **no `rsync`** (`rsync: command not found`; the host has `/usr/bin/rsync`, so the original "verified" sync was a manual host-shell run). Every automated night failed the INDIANCREEK leg (B2 + `dr-vault` legs kept succeeding); INDIANCREEK was stuck at `db_20260722` (~28h stale). **FIX (commit pending, live on NAS):** the offsite step now runs as a **pinned sibling container** `truxon-offsite-rsync:1` (built from repo `deploy/backup/offsite-rsync/Dockerfile` = alpine:3.21 + rsync + openssh-client + a uid-1000 `offsite` user), the same idiom as pg_dump (postgres:17-alpine) and B2 (amazon/aws-cli) — NOT apk-at-runtime. Two gotchas solved: (1) ssh rejects a private key not owned by the running uid → run `--user 1000:1000` AND bake uid-1000 into the image (bare `--user 1000` fails "No user exists for uid 1000" because ssh needs getpwuid); (2) the container has no `~/.ssh` trust store → host key **pinned** in `/volume1/docker/truxon-backup/.ssh/offsite_known_hosts` (seed: `ssh-keyscan -H "$OFFSITE_HOST" > .ssh/offsite_known_hosts`) with `StrictHostKeyChecking=yes`. Verified end-to-end 2026-07-23: `db_20260723` (77546753 B, exact byte-match) landed on INDIANCREEK in ~24s; signing leg green.
