**Built 2026-07-20 (commits 0462145 + 766534c, local only, not pushed).** [[project-truxon]] Customers section had many blank fields after the ITS import; Trux now backfills them from each customer's paperwork.

**How:** `customer-enrich` edge function (admin JWT or anon-bearer cron; verify_jwt off, gated in-function like [[qbo-integration]]'s qbo-sync) gathers each customer's docs (own attachments + recent load rate-cons, ≤3), runs the existing `extract-pdf` brain (`_shared/extract_llm.ts` unpdf + `customerPrompt`, moved to shared), and writes via the `apply_customer_enrichment` RPC. Frontend: admin "Fill blanks from documents" button on Customers.tsx (cursor-paged loop). Monthly `pg_cron` `truxon-customer-enrich-monthly` (`0 4 1 * *`) runs a maintenance sweep.

**Write policy — user chose "auto-fill blanks, no review" (2026-07-20):** the RPC is the SOLE write path and by construction fills ONLY empty columns, NEVER overwrites, and `company_name` is off the allow-list (identity protected). Every fill logged to `customer_enrichment_log` (field + value + source_document_id) + `customers.enriched_at` stamp. A **name-match guard** in the function only trusts a doc whose extracted broker name shares a significant token with the customer — so a mis-filed rate-con can't poison another customer. 12 pgTAP assertions cover all of this (suite 207/207).

**⚠ DOCUMENTS DON'T WORK HERE (measured 2026-07-20):** 0 customer-attached docs; all 2457 docs are load-level and **scanned images** (no text layer). unpdf can't read them AND grinds ~30-50s each (uncancellable) → blows the edge 150s cap. Sampled ~18 customers: 0 filled. Doc enrichment kept as a backup (blanks-only, added a 4MB/18s skip so it's not pathologically slow) but yields ~nothing on this corpus. Also: edge fns hard-cap ~150s, so per-invocation work must be tiny (limit≤6, 2 docs, internal per-customer deadline in runBatch).

**✅ QBO IS THE REAL SOURCE (built + RAN on prod 2026-07-20):** `qbo-sync` **mode:customers** (`syncCustomers`, `SELECT * FROM Customer`, match by qbo_id then name) pulls structured contact/address → same blanks-only RPC. **Live result: matched 154, filled 270 fields on 125 customers — billing 93 blank→1, email 231→118, contact 239→175.** Phone stays mostly blank (234): neither QBO nor scanned docs carry broker phones — honest data gap. Note: QBO Customer query rejects explicit `BillAddr` in SELECT list → must use `SELECT *`. The [[qbo-integration]] anon-key is the new non-JWT publishable format; the cron gate needs the **JWT-format** anon token (in the cron migrations) since it decodes role/ref.

**3rd source — vision rate-cons (built 2026-07-20):** the scanned rate cons DO contain contact info; read them with vision AI. `enrichCustomerFromRateCons` (data.ts) downloads a customer's loads' rate cons, renders pages in the browser (`renderPdfPages` — server can't rasterize PDFs in Deno), extracts via `extract-pdf` vision path, posts fields to customer-enrich's **apply_fields** branch (admin → service RPC, blanks-only, name-match guard). Reaches phone/contact QBO lacks. Client-side only (needs frontend deployed); rate-limited (extract-pdf 30/hr) so the "Scan rate confirmations (vision AI)" button pauses at the cap and resumes.

**Button** (Customers page): "Fill blanks (QuickBooks + documents)" = QBO then text-docs; separate "Scan rate confirmations (vision AI)" = the vision pass. Monthly crons: `truxon-customer-enrich-monthly` (docs, 04:00) + `truxon-qbo-customers-monthly` (QBO, 04:30) — both blanks-only.

**Confirmed 2026-07-21 the remaining ~104 blanks have NO other source:** dispatch inbox has no matching mail (0 mineable overlap); QBO stores these brokers as bare name-only records (all 13 QBO-linked blank-email customers returned null email, matched by exact qbo_id); FMCSA only 2/104 have an MC# and name-search is too fuzzy to auto-write. So the scanned rate cons (vision) are the ONLY source for these — hence the worker matters.

**Vision moved to GPU (Lynx) + FMCSA number-verification — 2026-07-22:** the NAS vision worker now points `OLLAMA_URL` at **[[gpu-box]] Lynx** (`http://100.110.143.84:11434`, model `qwen2.5vl:7b`) instead of the NAS CPU — ~10 s/doc vs minutes. `vision.env` `RASTER_DPI=150` (200 overflowed the 4096 ctx; `num_ctx=8192` OOMs the 8 GB card's KV cache — keep defaults). Still **on-demand only (no cron)**. Verified numbers are canonicalized; a verified MC back-fills a blank USDOT. Gated in `customer-enrich` (both write paths, deployed to prod) + no-op guards in qbo-sync/dispatch-watch. **`FMCSA_WEBKEY` is SET** (prod edge secret since 2026-07-20 — owner provided it) so the gate verifies against **QCMobile** (covers brokers), not just fail-closed. 9 pgTAP-style deno unit tests green.
