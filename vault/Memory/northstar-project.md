"Northstar" is Ilker's name for taking Truxon from Level 4 (connected + assistive + proactive) to **Level 5 — predictive**. Today the system reads/files/enriches/watches/nudges and learns from corrections, but it does NOT forecast or act autonomously (autonomy is deliberately off — shadow/propose-only).

**Approach (2 gears):** build transparent statistical models in Postgres FIRST (moving averages, per-entity historical distributions, simple regression — explainable, no cold-start on the ITS-backfilled tables), then graduate to real ML on the NAS (gradient-boosted trees, scored back into Postgres) only where many features beat a formula.

**The forecast backlog Ilker asked me to remember (ranked value × readiness):**
1. **Cash-flow forecast (4–8 wk)** — open AR + per-customer days-to-pay + booked/uninvoiced loads + recurring costs. SQL/stats. Data ready NOW. *Recommended first anchor.*
2. **Slow-pay risk per broker** — each customer's days-to-pay distribution → which open invoices will cross 30/60 before they do. Ready NOW.
3. **Load margin at booking** — lane/miles/rate/deadhead/fuel/driver-pay → predicted NET before accepting. Formula first, then ML on completed loads' actual net. The flagship. Ready NOW, improves live.
4. **Breakdown / big-repair risk per truck** — repair history + miles + age + repeat-reactive + MPG drift + FMCSA vehicle-OOS + (now) ELD engine hours/fault codes. True ML; the NAS #5 job. Needs a couple months of live data.
5. **Weekly revenue & utilization forecast** — built on the [[week-standard]] (Mon–Sun, same-week-last-year). Moving average + seasonality. Ready NOW.

**Recommended build order:** trio #1+#2+#5 first (all SQL/stats, shippable before Aug 1 go-live), then #3, then #4 when maintenance data thickens. Surfaces as a Dashboard Forecast card + a predictive upgrade to Sentinel ("this invoice WILL go late").
